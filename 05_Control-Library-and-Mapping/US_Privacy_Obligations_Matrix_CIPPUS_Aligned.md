# US Privacy Obligations Matrix (CIPP/US-aligned, practical)

**Purpose:** Translate US privacy expectations into vendor obligations that can be assessed and evidenced.

## Core US privacy expectations (practical control obligations)
| Obligation | Vendor obligation (what we require) | Evidence examples |
|---|---|---|
| Notice & transparency support | vendor provides clear processing description and data use | DPA annex, data flow |
| Purpose limitation | no use beyond contracted purpose | contract clause, policy |
| Data minimization | only required fields collected/processed | schemas, intake mapping |
| Retention & deletion | defined retention + deletion/return at end of service | retention policy, deletion cert |
| Access controls | least privilege for personnel accessing personal data | IAM policy, role controls |
| Service provider commitments | no “sale/share” outside purpose; subprocessor controls | contract terms, subprocessor list |
| Security safeguards | encryption, auth, logging, vuln mgmt | security evidence pack |
| Incident/breach readiness | IR plan + notification commitments + contacts | IR plan, SLA |
| Subprocessor governance | flow-down controls + change notification | contracts, notices |
| Data subject/consumer rights support (contractual) | ability to support deletion/access/correction requests when required by customer contract | workflow, ticket examples |

> Note: US laws vary by state and sector. This matrix is a defensible operational standard for vendor oversight and contracting.
